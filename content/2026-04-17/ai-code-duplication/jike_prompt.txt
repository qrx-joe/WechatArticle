# 同一个项目，我做了两遍



![文章封面：同一个项目，我做了两遍](images/image_00.png)

> 不是第一版不够好，是我以为"重来一次会更好"。

---

## 一、事情的起点

三个月前，我决定做一个"轻量级类 Coze 的智能体平台"。

那时候我主要在用两个 AI 编程助手：一个是 OpenAI 的 Codex，一个是 Anthropic 的 Claude。我突发奇想——**如果分别以这两个 AI 为主力，各自独立完成同一个需求，结果会有什么不同？**

听起来像是一个有趣的实验。于是我开了两个 GitHub 仓库，给它们下达了几乎完全一样的需求描述。AgentFlow-Studio 以 Codex 为主要开发模型，LOOM 以 Claude Code 为主要开发模型。两个项目都不是"纯靠一个模型从头到尾写出来的"——过程中我也会切换工具、手动调整、甚至让另一个模型帮忙 review——但主力工具确实是不同的。

三个月后，两个项目都"完成"了。但我坐在电脑前，看着两个仓库里高度重叠的代码，心情很复杂。

**我不是得到了两个好东西。我是得到了一个教训。**

---

## 二、第一版：以 Codex 为主的"务实主义"

第一版叫 AgentFlow-Studio，以 OpenAI 的 Codex 作为主力开发工具。

它的特点是：**先让我把东西跑起来。**

Vue 3 前端、NestJS 后端、PostgreSQL 数据库、Docker 部署——这些都在一周内搭了起来。它甚至给我写了 CI 流水线、pre-commit hooks、commitlint 规范。我 push 代码之后，GitHub Actions 会自动跑测试、检查格式、构建镜像。

工作流引擎能跑、知识库能上传文档、AI 对话能流式输出。功能上，它覆盖了我想做的 80%。

但问题也很明显。

代码结构有点乱。TypeScript 类型不够严格，有些地方用了 `any` 偷懒。CLAUDE.md（对，我让它写了一个）只有半页纸，简单交代了技术栈和目录结构。注释很少，有些地方我需要读三遍才能看懂逻辑。

**它的潜台词是："能用就行，别废话。"**

---

## 三、第二版：以 Claude Code 为主的"完美主义"

做完第一版之后，我心里有个声音在响："不够好。类型不严格、文档不够全、架构不够优雅。"

于是我又开了一个新仓库，这次换成以 Claude Code 作为主力工具。我给了更严格的约束。

结果，LOOM 诞生了。

Claude Code 的产出完全是另一种气质。它的 CLAUDE.md 写了整整三页：严格的 TypeScript 规范、命名约定、API 统一响应格式、绝对禁止事项清单、Definition of Done。它要求所有代码必须过 `strict: true`，不许用 `any`，公共方法必须声明返回类型。

它的架构也更"干净"。每个模块职责单一，代码分层清晰，还配了一份详细的 PRD 文档。

但当我兴奋地打开项目，准备体验这个"更优秀"的版本时，我发现了不对劲的地方。

**CI 流水线没了。**

**Husky 钩子没了。**

**Agent 编排模块没了。**

**工作流回滚机制也没了。**

LOOM 的 README 写得比 AgentFlow-Studio 更漂亮，文档更完整，代码规范更严格。但它缺失了第一版里那些真正跑在生产环境里的工程实践。

**它的潜台词是："规范必须完美，其他的以后再说。"**

---

## 四、当我把两个项目并排放在一起

这时候，有趣的事情发生了。

我打开两个仓库，并排放在一起对比。然后我发现了一个让我愣住的事实：

**两个项目的功能列表，重叠度超过 90%。**

它们都实现了：
- 可视化拖拽工作流
- 知识库 RAG 检索
- AI 对话（SSE 流式）
- 文档上传与向量化

它们用的技术栈几乎一样：Vue 3 + NestJS + TypeORM + 向量数据库。

**断断续续做了三个月，得到的不是两个互补的产品，而是同一个产品的两个 draft。**

更讽刺的是，在对比中我发现了第二版的一个"退化"：

AgentFlow-Studio 有 `.github/workflows`，LOOM 没有。AgentFlow-Studio 有 `compensation-executor` 回滚机制，LOOM 没有。AgentFlow-Studio 有 Redis 缓存模块，LOOM 选择了更轻量但也更简陋的方案。

**我用"更严格的规范"换来了"更残缺的工程实践"。**

这就像你重新装修房子，把墙面刷得更白、地板铺得更平整，但忘了装水电。

---

## 五、AI 不会告诉你的真相

很多人讨论"哪个 AI 编程更强"，但这个问题本身就是陷阱。

Codex 和 Claude Code 不是"谁更好"的问题，而是**作为主力工具，它们会把你推向不同的极端**。

以 Codex 为主力的开发过程倾向于"先完成再完美"。它更愿意写能让项目跑起来的代码，哪怕有些地方不够优雅。它的工程直觉更强——知道 CI、hooks、回滚机制这些"脏活累活"很重要。

以 Claude Code 为主力的开发过程倾向于"先定义再实现"。它更愿意花时间写规范、画架构、建约束。它的代码更整洁，文档更完整，但有时会陷入"规范先行"的陷阱——写了一大堆约束，却没有给工程实践留位置。

**问题在于：不是 AI 选择了这些风格，是 AI 放大了你潜意识里已经存在的偏好。**

当我以 Codex 为主力做第一版时，我内心想的是"先做出来看看"。当我以 Claude Code 为主力做第二版时，我内心想的是"这次一定要做好"。两个 AI 只是忠实地执行了我没说出口的真实需求。

---

## 六、真正的浪费不是代码，是"不敢承认"

两个项目做完之后，我面临一个选择：

**删掉一个，还是继续维护两个？**

删掉意味着承认"三个月里有一个半月是白费的"。继续维护两个，意味着接受双倍的依赖更新、双倍的 bug 修复、双倍的文档维护。

大多数人会选择后者。因为"已经投入了那么多，不能浪费"。

这就是**沉没成本谬误**。你不是因为项目有价值而继续维护它，你是因为不想承认自己曾经浪费了时间。

我最终的决定是：**开一个新仓库，把两个项目的优点合并在一起。**

把 AgentFlow-Studio 的 CI、回滚机制、Redis 缓存、Agent 编排搬到 LOOM 的代码基础上。原来的两个仓库保留不动——毕竟是花了心血的"实验记录"，留着自己看，也留给想围观的人。


---

## 七、我学到的东西

**第一，"重来一次会更好"是最大的幻觉。**

你不是因为第一版不够好才想重做，你是因为**看到了第一版的所有问题**。但任何项目写完之后都会有问题。重做一遍只会让你看到新版的问题，而不是消除问题。

**第二，规范和工程实践是两件事。**

LOOM 的 CLAUDE.md 写了三页规范，但缺失了 CI 和 hooks。这说明**文档上的完美不等于工程上的完整**。规范是让代码好看，工程实践是让代码能活。

**第三，AI 编程最大的风险不是代码质量，是你自己的完美主义。**

两个 AI 都写得不错，但我在中间做了一个错误的决策："我觉得可以更好，所以重做一遍。"AI 没有逼我开第二个仓库，是我自己的完美主义逼的。

**第四，"够好"比"完美"更有价值，"能用"比"重写"更难得。**

AgentFlow-Studio 的代码不够优雅，但它能跑、有 CI、有回滚。LOOM 的代码更优雅，但缺失了这些基础设施。**优雅是奢侈品，可用是必需品。**

---

## 八、写在最后

两个项目都已经开源了。如果你有兴趣，可以去看看我的"重复建设现场"。

但我不希望你从我的代码里学到什么技术。我希望你从这个故事里看到一样东西：

> **当你想删掉一个旧项目、重写一个新项目的时候，先问自己一个问题：你到底是发现了"旧项目无法解决的问题"，还是只是"看不惯旧项目的写法"？**

如果是后者，停下来。去把旧项目的 CI 修好，去把注释补上，去把那个丑陋的函数重构一下。

**迭代比重写更勇敢。承认"够好了"比追求"完美"更需要智慧。**

这才是我做了两遍同一个项目之后，真正想告诉你的事情。

## 即刻平台特点
- 内容长度：200-400 字最佳（主帖）
- 格式：纯文本+图，不需要复杂排版
- 阅读场景：信息流刷过，3秒定生死，开头必须有钩子
- 不需要标题，开头第一句就是标题
- 标签很重要，决定推荐流量，必须加相关标签

## 输出要求

### 1. 主帖文案
- 用第一人称，语气真实自然
- 开头一句必须是强钩子（悬念、冲突、反常识）
- 主体 200-400 字，保留最核心的故事线
- 结尾要有情绪落点或行动号召
- 添加 3-5 个相关标签（格式：#标签名）

### 2. 评论区补充（可选）
- 如果主帖内容较长，可以拆分成 2-3 条评论补充
- 每条评论 50-150 字
- 评论可以展开细节、解释背景、或引导互动

### 3. 配图建议
- 说明主帖配哪张图（从文章配图方案中选择）
- 即刻信息流中显示为正方形缩略图

### 4. 发布时机建议
- 最佳发布时间：工作日晚上 21:00-23:00

## 输出格式

请按以下格式输出：

```
## 主帖
[主帖文案]

## 评论1
[评论内容]

## 评论2
[评论内容]

## 标签建议
#标签1 #标签2 #标签3

## 配图建议
主帖配图：[图片文件名]
```
